# Feature Specification: 會員登入

**Feature Branch**: `004-Login`

**Created**: 2026-05-27

**Status**: Ready for Planning

**Input**: 新增會員登入功能，前端實作一個簡單的登入表單，包含電子信箱和密碼欄位，並將資料送到後端 API 進行驗證。後端 API 接收登入請求，驗證電子信箱和密碼是否正確，並回傳登入成功或失敗的結果。需使用 JWT（JSON Web Token）來管理登入狀態，確保安全性。前端在登入成功後，將 JWT 儲存在安全的地方（如 HttpOnly Cookie 或 Local Storage），並在後續的 API 請求中附帶 JWT 以驗證使用者身份。若閒置時間過長，JWT 可能會過期，前端應該能夠處理過期的情況，提示使用者重新登入。

---

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 訪客完成會員登入 (Priority: P1)

一位已註冊的會員，在 MiraiShop 網站填寫電子信箱和密碼，送出後系統驗證資料正確性，並回傳登入成功或失敗的結果。登入表單需提供前往註冊頁面的連結，讓新訪客可以直接前往註冊。

**Why this priority**: 這是整個會員功能的基礎。沒有這個流程，使用者無法登入、無法享有購物、訂單追蹤等服務。

**Independent Test**: 前端提供簡單的登入表單，後端 API 接收登入請求並驗證資料。測試時可使用已註冊的會員資料進行登入，確認系統正確回應。

**Acceptance Scenarios**:

1. **Given** 訪客位於登入頁面，**When** 正確填寫電子信箱和密碼並送出，**Then** 系統顯示「登入成功」，且會員資料已驗證，並取得可用的存取憑證。
2. **Given** 訪客填寫的電子信箱或密碼不正確，**When** 送出表單，**Then** 系統顯示「電子信箱或密碼錯誤」，表單保留已填內容，不揭露哪個欄位有誤。
3. **Given** 訪客未填寫某個必填欄位，**When** 試圖送出，**Then** 系統在該欄位下方顯示提示訊息，不送出資料。
4. **Given** 訪客位於登入頁面，**When** 點擊「前往註冊」連結，**Then** 系統導向會員註冊頁面。

---

### User Story 2 - 系統拒絕不完整的資料 (Priority: P2)

訪客嘗試在欄位未填寫完整或格式有誤的情況下送出登入表單，系統應在客戶端即時攔截，顯示對應欄位的錯誤提示，不向伺服器發出請求，確保資料品質並減少不必要的網路消耗。

**Why this priority**: 前端驗證可即時回饋使用者，提升操作體驗；同時避免無效請求送達後端，保護系統資源。

**Independent Test**: 直接開啟登入表單，分別嘗試：空白表單送出、僅填電子信箱、僅填密碼、填入格式錯誤的電子信箱，確認各情境均顯示對應提示且未送出 API 請求。

**Acceptance Scenarios**:

1. **Given** 訪客位於登入頁面且未填寫任何欄位，**When** 點擊送出，**Then** 系統在電子信箱及密碼欄位下方各自顯示「此欄位為必填」，不發送請求。
2. **Given** 訪客僅填寫電子信箱但未填寫密碼，**When** 點擊送出，**Then** 系統在密碼欄位下方顯示「此欄位為必填」，不發送請求。
3. **Given** 訪客填寫格式不正確的電子信箱（如缺少 @），**When** 點擊送出，**Then** 系統在電子信箱欄位下方顯示「請輸入有效的電子信箱格式」，不發送請求。

---

### User Story 3 - 已登入會員的存取憑證過期處理 (Priority: P3)

一位已登入的會員，在閒置一段時間後回到頁面繼續操作，系統偵測到登入憑證已過期，主動提示使用者重新登入，並引導至登入頁面，不呈現未預期的錯誤畫面。

**Why this priority**: 存取憑證過期是正常的安全機制，妥善的過期提示能避免使用者困惑，提升整體信任感。

**Independent Test**: 以短效期的測試憑證登入，等待過期後嘗試執行需要驗證的操作，確認系統顯示過期提示並導向登入頁面。

**Acceptance Scenarios**:

1. **Given** 已登入會員的存取憑證已過期，**When** 嘗試執行需驗證的操作，**Then** 系統顯示「登入逾時，請重新登入」並跳轉至登入頁面。
2. **Given** 會員重新登入後，**When** 驗證成功，**Then** 系統恢復正常操作狀態，取得新的存取憑證。

---

### Edge Cases

- 密碼欄位在畫面上應全程遮蔽，不得以明文顯示。
- 同一來源在短時間內大量重複送出登入請求，系統應有基本的速率限制機制加以防護。
- 密碼在儲存時須加入鹽值（salt）以防範彩虹表攻擊（利害關係人指定使用 SHA-256 演算法，應搭配 salt 使用）。
- 電子信箱格式驗證應同時在前端與後端進行，防止繞過前端驗證的惡意請求。

---

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 系統 MUST 提供一個會員登入表單，包含以下必填欄位：電子信箱（Email）、密碼。
- **FR-002**: 系統 MUST 在使用者送出表單前，於客戶端驗證所有必填欄位均已填寫且 Email 格式正確。
- **FR-003**: 系統 MUST 在後端驗證電子信箱與密碼的組合是否正確；若不符合，回傳統一錯誤訊息，不揭露哪個欄位有誤。
- **FR-004**: 系統 MUST 在儲存及比對密碼時，以加鹽（salt）雜湊的方式處理，確保資料庫中不存在明文密碼。
- **FR-005**: 後端 API MUST 以結構化的資料格式（DTO）接收前端送出的登入資料，並完成驗證後回應。
- **FR-006**: 登入成功後，系統 MUST 簽發一個含有效期的存取憑證，供後續 API 請求驗證使用者身份。
- **FR-007**: 系統 MUST 在偵測到存取憑證過期或無效時，回應未授權狀態；前端收到此狀態後應提示使用者重新登入。
- **FR-008**: 登入表單 MUST 提供前往會員註冊頁面的連結。

### Key Entities

- **Member（會員）**: 代表一位已完成註冊的使用者。核心屬性：唯一電子信箱（作為識別碼）、姓名、通訊地址、住址、加鹽雜湊後的密碼、註冊時間。
- **LoginRequest（登入請求）**: 前端送出的資料結構，包含：電子信箱、密碼（明文，傳送後由後端加密比對）。
- **AccessToken（存取憑證）**: 登入成功後簽發的憑證，包含：會員識別碼、簽發時間、有效期限。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 使用者在正常網路環境下，從開啟登入頁面到完成送出，整體流程可在 3 分鐘內完成。
- **SC-002**: 系統在收到有效的登入請求後，於 2 秒內回應成功或失敗的結果。
- **SC-003**: 表單驗證錯誤訊息在使用者送出的當下即顯示，不需等待伺服器回應。
- **SC-004**: 90% 的使用者在無說明文件的情況下，能在第一次嘗試時成功完成登入。
- **SC-005**: 資料庫中的密碼欄位，100% 不得以明文形式存在。
- **SC-006**: 存取憑證過期時，系統能在使用者操作 1 秒內呈現重新登入的提示。

---

## Assumptions

- 電子信箱採用**電子信箱（Email）**作為唯一識別碼，系統須驗證 Email 格式正確性。
- 前端表單與後端 API 之間透過結構化的資料物件（DTO）傳遞資料，後端負責所有安全性驗證。
- 登入成功後前端以安全方式存放存取憑證，具體存放機制（HttpOnly Cookie 或 Local Storage）依安全政策決定。
- 密碼雜湊演算法採用利害關係人指定的 SHA-256，並加入鹽值（salt）以提升安全性。
- 登入成功後的頁面跳轉目標（首頁或原訪問頁面）超出本次功能範圍，於後續版本定義。
- 本功能僅涵蓋電子信箱與密碼的登入流程，不包含第三方登入（Google、Facebook 等）。
- 速率限制的具體閾值（如每分鐘最多嘗試次數）依系統運維政策決定，本規格僅要求機制存在。
