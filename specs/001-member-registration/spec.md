# Feature Specification: 會員註冊

**Feature Branch**: `001-member-registration`

**Created**: 2026-05-27

**Status**: Ready for Planning

**Input**: 新增會員註冊功能,前端必填為:姓名、地址(地址有兩個,通訊地址以及住址)、密碼(須加密存資料庫,使用SHA-256),後端API接收前端回傳的DTO

---

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 訪客完成會員註冊 (Priority: P1)

一位想要購物的訪客，在 MiraiShop 網站填寫必要的個人資料（姓名、通訊地址、住址、電子信箱、密碼），送出後系統建立會員電子信箱，並讓該訪客以新電子信箱的身份使用後續功能。

**Why this priority**: 這是整個會員功能的基礎。沒有這個流程，使用者無法成為會員、無法享有購物、訂單追蹤等服務。

**Independent Test**: 可以單獨開啟註冊頁面，填寫所有必填欄位並提交，確認系統回應「註冊成功」且資料庫中已建立該筆會員紀錄。

**Acceptance Scenarios**:

1. **Given** 訪客位於註冊頁面，**When** 正確填寫所有必填欄位（姓名、通訊地址、住址、電子信箱、密碼）並送出，**Then** 系統顯示「註冊成功」，且會員資料已儲存。
2. **Given** 訪客填寫的電子信箱已被其他人使用，**When** 送出表單，**Then** 系統顯示「此電子信箱已被使用，請換一個」，表單保留已填內容。
3. **Given** 訪客未填寫某個必填欄位，**When** 試圖送出，**Then** 系統在該欄位下方顯示提示訊息，不送出資料。

---

### User Story 2 - 系統拒絕不完整的資料 (Priority: P2)

當訪客未填寫所有必填欄位或填寫格式不正確時，系統應即時告知，避免無效資料進入系統。

**Why this priority**: 資料完整性保護，確保每一筆會員資料都符合最低要求，降低後續異常。

**Independent Test**: 分別留白每一個必填欄位送出，確認系統每次都在對應欄位顯示明確的錯誤提示。

**Acceptance Scenarios**:

1. **Given** 任何必填欄位為空白，**When** 送出表單，**Then** 系統在該欄位旁顯示「此欄位為必填」。
2. **Given** 通訊地址或住址填寫過短（少於 5 個字），**When** 送出，**Then** 系統顯示「地址格式不正確」。

---

### Edge Cases

- 訪客使用已存在的電子信箱嘗試重複註冊，系統應拒絕並提示電子信箱重複。
- 姓名或地址欄位含有特殊字元（如 `<script>`），系統應拒絕或清除，不得儲存有害內容。
- 密碼欄位在畫面上應全程遮蔽，不得以明文顯示。
- 同一 IP 在短時間內大量重複送出註冊，系統應有基本的防護機制。

---

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 系統 MUST 提供一個會員註冊表單，包含以下必填欄位：姓名、通訊地址、住址、電子信箱（Email）、密碼。
- **FR-002**: 系統 MUST 在使用者送出表單前，驗證所有必填欄位均已填寫。
- **FR-003**: 系統 MUST 拒絕已存在的電子信箱重複註冊，並回傳明確的錯誤訊息。
- **FR-004**: 系統 MUST 在儲存會員資料前，將密碼以單向不可逆的方式加密，確保資料庫中不存在明文密碼。
- **FR-005**: 後端 API MUST 以結構化的資料格式（DTO）接收前端送出的註冊資料，並完成驗證後儲存。
- **FR-006**: 系統 MUST 在所有欄位驗證通過後，回傳明確的成功回應。
- **FR-007**: 系統 MUST 在任何欄位驗證失敗時，回傳對應的錯誤說明，讓前端可以呈現給使用者。
- **FR-008**: 系統 MUST 分開儲存通訊地址與住址，兩者為獨立欄位。

### Key Entities

- **Member（會員）**: 代表一位已完成註冊的使用者。核心屬性：唯一電子信箱（作為電子信箱）、姓名、通訊地址、住址、加密後的密碼、註冊時間。
- **RegistrationRequest（註冊請求）**: 前端送出的資料結構，包含：姓名、通訊地址、住址、電子信箱、密碼（明文，傳送後由後端加密）。

---

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 使用者在正常網路環境下，從開啟註冊頁面到完成送出，整體流程可在 3 分鐘內完成。
- **SC-002**: 系統在收到有效的註冊請求後，於 2 秒內回應成功或失敗的結果。
- **SC-003**: 表單驗證錯誤訊息在使用者送出的當下即顯示，不需等待伺服器回應。
- **SC-004**: 90% 的使用者在無說明文件的情況下，能在第一次嘗試時成功完成註冊。
- **SC-005**: 資料庫中的密碼欄位，100% 不得以明文形式存在。

---

## Assumptions

- 電子信箱採用**電子信箱（Email）**作為唯一識別碼，同時作為未來登入的憑證，系統須驗證 Email 格式正確性。
- 密碼加密使用 SHA-256 演算法（此為專案關係人指定之技術約束）。注意：SHA-256 作為密碼儲存方案在業界有安全疑慮（無鹽值設計容易遭彩虹表攻擊），建議後續評估是否加入鹽值（salt）。
- 前端表單與後端 API 之間透過結構化的資料物件（DTO）傳遞資料，後端負責所有安全性驗證。
- 地址欄位為純文字輸入，不串接地址查詢 API（如郵遞區號自動補完），可於後續版本擴充。
- 註冊成功後的下一步（如是否自動登入、是否發送驗證信）超出本次功能範圍，於後續版本定義。
- 本功能僅涵蓋「新訪客建立電子信箱」，不包含第三方登入（Google、Facebook 等）。
